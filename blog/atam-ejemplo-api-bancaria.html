ATAM in Practice: A Banking Movements API

A complete ATAM analysis on a REST API for querying banking account movements. Utility tree, architectural approaches, tradeoffs, and documented risks.

May 2026 Felipe Pabón

En el artículo anterior vimos qué es ATAM y cuándo usarlo. Ahora vamos a aplicarlo sobre un sistema concreto: un API REST de consulta de movimientos de cuenta bancaria.

Este es el tipo de sistema que parece simple desde afuera — "devuelve una lista de transacciones" — pero que esconde tensiones arquitectónicas serias: seguridad vs. rendimiento, disponibilidad vs. consistencia, flexibilidad vs. cumplimiento regulatorio. ATAM hace visibles exactamente esas tensiones.

In the previous article we covered what ATAM is and when to use it. Now let's apply it to a concrete system: a REST API for querying banking account movements.

This is the kind of system that looks simple from the outside — "return a list of transactions" — but hides serious architectural tensions: security vs. performance, availability vs. consistency, flexibility vs. regulatory compliance. ATAM makes those tensions visible.

El sistema

El API expone un endpoint que permite a los clientes del banco consultar los movimientos de sus cuentas. Es consumido por la app móvil y el portal web, y se conecta a un core bancario legado para obtener los datos.

The System

The API exposes an endpoint that allows bank customers to query their account movements. It's consumed by the mobile app and web portal, and connects to a legacy core banking system to retrieve data.

Drivers del negocio

  • Disponibilidad 24/7 — los clientes acceden desde cualquier zona horaria
  • Cumplimiento PCI-DSS y normativa bancaria local
  • SLA de respuesta: p95 < 500ms bajo carga normal
  • Dependencia del core bancario legado con ventanas de mantenimiento nocturnas
  • Obligación de mantener logs de auditoría por 90 días mínimo

Business Drivers

  • 24/7 availability — customers connect from any timezone
  • PCI-DSS compliance and local banking regulation
  • Response SLA: p95 < 500ms under normal load
  • Dependency on legacy core banking with nightly maintenance windows
  • Mandatory audit log retention for at least 90 days

Enfoques Arquitectónicos

Cuatro decisiones de diseño con impacto directo sobre los atributos de calidad identificados:

EA-1: Autenticación con OAuth2 + JWT

Decisión: El API Gateway valida tokens JWT firmados por un servidor de autorización OAuth2. El payload del token incluye el ID de cuenta, scopes y expiración.

Atributos satisfechos: Seguridad (Seg-1), cumplimiento PCI-DSS.

Atributos en tensión: Rendimiento (validación del token añade ~20ms), Disponibilidad (si el servidor de autorización falla, el sistema queda inutilizable).

Alternativas descartadas: API Keys estáticas (difíciles de revocar), sesiones en servidor (no escalan horizontalmente).

EA-2: Cache Redis con TTL corto (30 segundos)

Decisión: Las respuestas del core bancario se cachean en Redis por 30 segundos. Las consultas repetidas dentro de ese ventana no llegan al core.

Atributos satisfechos: Rendimiento (Ren-1), Disponibilidad durante mantenimiento (Dis-1).

Atributos en tensión: Consistencia — un movimiento reciente puede tardar hasta 30 segundos en aparecer.

Alternativas descartadas: Sin cache (core no soporta la carga), TTL largo (inaceptable para datos financieros).

EA-3: Circuit Breaker hacia el core bancario

Decisión: Si el core bancario falla más del 50% de las solicitudes en una ventana de 10 segundos, el circuit breaker se abre y el sistema sirve exclusivamente desde cache.

Atributos satisfechos: Disponibilidad (Dis-1, Dis-2).

Atributos en tensión: Consistencia — en estado abierto, los datos son definitivamente obsoletos. Sin mitigación visible al usuario.

Alternativas descartadas: Retry infinito (agrava la sobrecarga del core), timeout simple (no previene efecto cascada).

EA-4: Logs de auditoría append-only en almacenamiento separado

Decisión: Cada consulta genera una entrada de auditoría (usuario, cuenta consultada, timestamp, IP) escrita en un store separado de solo escritura, inmutable por diseño.

Atributos satisfechos: Seguridad (Seg-2), cumplimiento regulatorio.

Atributos en tensión: Rendimiento — la escritura síncrona de auditoría añade latencia a cada operación.

Alternativas descartadas: Logs en la misma BD (modificables, no cumplen con auditoría), escritura asíncrona (riesgo de pérdida de registros ante falla).

Architectural Approaches

Four design decisions with direct impact on the identified quality attributes:

EA-1: Authentication with OAuth2 + JWT

Decision: The API Gateway validates JWT tokens signed by an OAuth2 authorization server. The token payload includes account ID, scopes, and expiration.

Attributes satisfied: Security (Sec-1), PCI-DSS compliance.

Attributes in tension: Performance (token validation adds ~20ms), Availability (if the auth server fails, the system becomes unusable).

Discarded alternatives: Static API keys (hard to revoke), server-side sessions (don't scale horizontally).

EA-2: Redis Cache with Short TTL (30 seconds)

Decision: Core banking responses are cached in Redis for 30 seconds. Repeated queries within that window don't reach the core.

Attributes satisfied: Performance (Perf-1), Availability during maintenance (Av-1).

Attributes in tension: Consistency — a recent movement may take up to 30 seconds to appear.

Discarded alternatives: No cache (core can't handle the load), longer TTL (unacceptable for financial data).

EA-3: Circuit Breaker to Core Banking

Decision: If the core banking system fails more than 50% of requests in a 10-second window, the circuit breaker opens and the system serves exclusively from cache.

Attributes satisfied: Availability (Av-1, Av-2).

Attributes in tension: Consistency — in open state, data is definitively stale. No visible indication to the user.

Discarded alternatives: Infinite retry (worsens core overload), simple timeout (doesn't prevent cascade failures).

EA-4: Append-Only Audit Logs in Separate Storage

Decision: Each query generates an audit entry (user, queried account, timestamp, IP) written to a separate write-only, immutable-by-design store.

Attributes satisfied: Security (Sec-2), regulatory compliance.

Attributes in tension: Performance — synchronous audit writes add latency to every operation.

Discarded alternatives: Logs in the same DB (modifiable, fail audit requirements), async writes (risk of record loss on failure).

Reporte Resumido

El análisis identificó 2 puntos de sensibilidad, 3 tradeoffs, 3 riesgos y 2 no-riesgos. Las recomendaciones priorizadas:

🔴 Crítico

  • R-1 — Implementar invalidación de cache por evento. Cuando el core emita un evento de reversión, invalidar proactivamente la entrada de cache afectada. Usar un event bus o webhook del core.
  • R-2 — Implementar revocación de JWT. Introducir una blacklist de tokens con TTL en Redis. Al logout o detección de anomalía, agregar el JTI a la blacklist.

🟡 Importante

  • R-3 — Añadir modo de mantenimiento planificado al circuit breaker. Permitir que operaciones configuren el breaker en modo "mantenimiento" para evitar aperturas no controladas durante ventanas conocidas.
  • TR-2 — Indicador visible de datos obsoletos. Cuando el circuit breaker esté abierto, agregar un header o campo en la respuesta que indique que los datos provienen de cache y pueden no ser recientes.

🟢 A considerar

  • TR-3 — Evaluar escritura asíncrona de auditoría con garantía de entrega. Si el SLA de 500ms se convierte en problema bajo alta carga, migrar el log de auditoría a escritura asíncrona con un broker de mensajes durable (ej: Kafka) que garantice no pérdida.
Los tradeoffs no son errores de diseño — son decisiones conscientes. El objetivo de ATAM es documentarlos para que el equipo sepa exactamente qué está apostando con cada elección.

Summary Report

The analysis identified 2 sensitivity points, 3 tradeoffs, 3 risks, and 2 non-risks. Prioritized recommendations:

🔴 Critical

  • R-1 — Implement event-driven cache invalidation. When the core emits a reversal event, proactively invalidate the affected cache entry. Use an event bus or core webhook.
  • R-2 — Implement JWT revocation. Introduce a Redis-backed token blacklist with TTL. On logout or anomaly detection, add the JTI to the blacklist.

🟡 Important

  • R-3 — Add planned maintenance mode to the circuit breaker. Allow operations to configure the breaker in "maintenance" mode to avoid uncontrolled trips during known windows.
  • TR-2 — Visible stale data indicator. When the circuit breaker is open, add a header or response field indicating that data comes from cache and may not be current.

🟢 To Consider

  • TR-3 — Evaluate async audit writes with delivery guarantee. If the 500ms SLA becomes a problem under high load, migrate audit logging to async writes with a durable message broker (e.g., Kafka) that guarantees no data loss.
Tradeoffs are not design mistakes — they are conscious decisions. The goal of ATAM is to document them so the team knows exactly what they're betting on with each choice.
